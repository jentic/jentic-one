# SDD Lessons

Recurring lessons learned from implementation that future spec authors should know about. Loaded by `/sdd-new-spec` and `/sdd-new-phase` before scaffolding so new specs absorb the learning.

This file is **not part of the constitution** (the constitution is mission + tech-stack + roadmap — see `.claude/rules/sdd-constitution.md`). It is an operational, append-only checklist that grows as the team ships phases.

## Lifecycle

- **Captured** in `specs/<date>-<slug>/retrospective.md` after an implementation PR surfaces a spec gap. The retrospective is human-written; `/sdd-implement-spec` only emits raw material in its PR body's `## Spec deviations` section.
- **Distilled** into this file by `/sdd-distill-lessons` (manual cadence): the skill reads all retrospectives, deduplicates, proposes additions via `AskUserQuestion`, writes on confirmation.
- **Graduated** to `specs/tech-stack.md` (manual edit) when a lesson recurs across multiple specs and names a load-bearing invariant rather than an empirical reminder. `/sdd-distill-lessons` flags promotion candidates; the promotion itself is a hand-rolled edit.

## Lessons

- When a spec adds a new path to the auth chain, `plan.md` must explicitly enumerate precedence against every existing path — including the multi-credential matrix (e.g., what happens when a request carries both `tk_` and the new header) — not just the insertion point. Express the two-actor invariant as an ordering proof, not a label. — captured from `specs/2026-05-12-trusted-proxy-forwarded-identity/retrospective.md`
- When a schema change introduces a new row type into a previously single-row table, `plan.md` must include a "grep `FROM <table>` and triage every call site" step. Patching only the path that introduces the new rows is not enough — legacy reads carry the old "one row exists" assumption. — captured from `specs/2026-05-12-trusted-proxy-forwarded-identity/retrospective.md`
- Adding a column to an existing table requires explicit migration sub-steps: (a) backfill for existing rows when the column has documented semantic values; (b) cleanup of rows that would violate any constraint being restored by downgrade. Without these, the column is unreliable in production and the downgrade is unsafe once new-shape rows exist. — captured from `specs/2026-05-12-trusted-proxy-forwarded-identity/retrospective.md`
- "No UI changes" is a claim about UI *code*, not UI *flow*. When a phase changes how users get into the system, audit every SPA surface whose **control signal** is a backend endpoint touched (or made meaningless) by the phase — `/health` drives the account-creation wizard, `/user/me` drives `AuthGuard`. Name those flows in `requirements.md` and decide explicitly whether they need a code change, a backend-signal change, or are genuinely unaffected. — captured from `specs/2026-05-12-trusted-proxy-forwarded-identity/retrospective.md`
- Before scaffolding finishes, cross-check `requirements.md` Decisions against `plan.md` env-gate conditions — they share the same vocabulary and must agree, otherwise docs and code drift along whichever side the implementer follows. — captured from `specs/2026-05-12-trusted-proxy-forwarded-identity/retrospective.md`
- Plan pseudocode is documentation, not paste-ready code. Describe behaviour ("read the ASGI peer IP defensively, treating absent or `None` as unknown"); leave the safe-access idiom to the implementer. Phase 28's literal `scope.get("client", ("", 0))[0]` looked safe but crashed on ASGI's present-but-`None` case. — captured from `specs/2026-05-12-trusted-proxy-forwarded-identity/retrospective.md`
- When a phase changes *who counts as a human admin*, `requirements.md` and the user-facing `docs/auth.md` must state the privilege scope in prose, not just `is_admin = True` in plan pseudocode. "Mirrors today's behaviour" hides the fan-out when today is a single-root-operator model and tomorrow is "everyone in the SSO directory". Operators must see the trade-off before they configure the env vars. — captured from `specs/2026-05-12-trusted-proxy-forwarded-identity/retrospective.md`
- When a phase changes session-token shape (JWT claims added/removed/renamed, cookie format, bearer-token encoding), the spec must include an **in-flight token compatibility** clause: tokens already in users' cookies survive the deploy, and the receiving code must either tolerate the old shape (graceful fallback) or force re-login (cookie invalidation on deploy). Call it out explicitly in `requirements.md` Decisions. — captured from `specs/2026-05-12-trusted-proxy-forwarded-identity/retrospective.md`
- When a spec decision changes between the scaffold and the implementation PRs (e.g. the build-system approach), update the roadmap phase body to match — `/sdd-new-spec` only scaffolds the spec dir; the roadmap body is not auto-synced. — captured from `specs/2026-05-20-migrate-pdm-to-uv/retrospective.md`
- When writing a grep-based "no references remain" validation check for a migration, include the current-phase spec dir in the exclusion list — it will naturally contain the old names as descriptive context. — captured from `specs/2026-05-20-migrate-pdm-to-uv/retrospective.md`
- Before specifying argument-passing syntax for a task runner (poe, just, make), verify the runner's actual interpolation syntax rather than carrying over syntax from the tool being replaced. — captured from `specs/2026-05-20-migrate-pdm-to-uv/retrospective.md`
